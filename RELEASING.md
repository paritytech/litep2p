# Release Checklist

Both stages of a litep2p release are automated:

- **Release Prep** (`.github/workflows/release-prep.yml`) — opens the release PR for you (bumps `Cargo.toml`, regenerates `Cargo.lock`, drafts a CHANGELOG entry from the PRs merged since the last release tag, runs `cargo publish --dry-run` against the result).
- **Release Publish** (`.github/workflows/release-publish.yml`) — a manually-triggered workflow that runs the final test suite, tags `master`, publishes to crates.io, and creates the GitHub Release. The publish step is gated by the `release` environment and requires reviewer approval.

The steps below assume those workflows are healthy. A manual fallback is documented at the bottom.

These do not need to be repeated for subsequent releases.

## Release steps

1. Ensure that everything you'd like to see released is on the `master` branch.

2. Go to the [Release Prep workflow](https://github.com/paritytech/litep2p/actions/workflows/release-prep.yml), click **Run workflow**, and supply the new `version` (e.g. `0.15.0`, no leading `v`). If unsure what to bump to, check with the Parity Networking team.

    The workflow will:
    - Bump `Cargo.toml` and regenerate `Cargo.lock`.
    - Prepend a `## [X.Y.Z] - YYYY-MM-DD` section to `CHANGELOG.md` listing every PR merged since the last release tag (generated by `.github/scripts/generate-litep2p-changelog.sh`). The list is **flat / unsorted** — bucketing is left to the reviewer.
    - Run `cargo publish --dry-run` to verify the crate packages cleanly.
    - Push a `release-vX.Y.Z` branch and open a PR titled `chore: Release litep2p vX.Y.Z`, whose body contains the same changelog list.

    Note: Release Prep is **create-only**. If the `release-vX.Y.Z` branch already exists it fails fast (see "If something goes wrong").

3. Review the auto-opened PR and **commit your changelog edits to the release branch**:
    - Re-bucket the entries into `### Added` / `### Changed` / `### Fixed` to match the existing CHANGELOG style.
    - Add a summary paragraph at the top of the new `## [X.Y.Z]` section.
    - Confirm `Cargo.toml` and `Cargo.lock` look right.

    **If CI hasn't started on the PR**, push an empty commit to trigger it (GitHub does not run `pull_request` workflows on PRs opened by `GITHUB_TOKEN`):

    ```bash
    git fetch origin && git checkout release-v0.15.0
    git commit --allow-empty -m "trigger ci" && git push
    ```

4. Once the PR is reviewed and CI is green, merge it.

5. Go to the [Release Publish workflow](https://github.com/paritytech/litep2p/actions/workflows/release-publish.yml), click **Run workflow**, and supply the **same `version`** you just merged.

    - The `verify` job runs the full test suite and checks that the requested version matches `Cargo.toml` on `master`. Review its result before approving the next step.
    - The `publish` job is gated by the `release` environment. Once `verify` passes, GitHub requests a reviewer's approval; once approved it **creates and pushes the `vX.Y.Z` tag on `master`**, publishes to crates.io, and creates a GitHub Release using the changelog section from step 3.

    You no longer create or push the tag by hand — the publish workflow does it. The tag is an annotated (unsigned) tag created by `github-actions[bot]`.

## If something goes wrong

- **Release Prep fails at `cargo publish --dry-run`** — the crate would fail to publish in its current state (e.g. a `Cargo.toml` field is missing, an `include`/`exclude` path is broken, or a path-dependency was left in). Fix the issue on `master` via a normal PR, then re-run Release Prep.
- **Release Prep fails with "Branch release-vX.Y.Z already exists on origin"** — the branch is left over from a previous run or has manual edits. To iterate on the pending release, edit that branch / its PR directly rather than re-running. To regenerate from `master`, delete the branch first (`git push --delete origin release-vX.Y.Z`) and re-run.
- **`verify` fails with "Requested version does not match Cargo.toml on master"** — you're publishing before the release-prep PR merged, or passed the wrong version. Merge the prep PR (so `master`'s `Cargo.toml` carries the new version), then re-run Release Publish with the matching version.
- **`cargo publish` fails with "crate version X.Y.Z is already uploaded"** — the publish already happened in a previous run. Tag creation and the GitHub Release step are both idempotent (the tag is skipped if it exists, the release is create-or-edit), so re-running after fixing an unrelated failure is safe; if the crate is genuinely already published you'll need to bump to a new patch version.
- **The publish workflow never asks for approval** — confirm the `release` environment exists and has required reviewers set; without that the publish job runs as soon as `verify` passes.

## Manual fallback

If either workflow is broken and a release is urgent, the prep can be done by hand:

1. Branch off `master` as `release-vX.Y.Z`.
2. Bump the `version` field in the root `Cargo.toml`.
3. Run `cargo generate-lockfile`.
4. Add a `## [X.Y.Z] - YYYY-MM-DD` section to `CHANGELOG.md` (see existing entries for format — the heading must match exactly for the publish workflow's release-notes extractor). You can generate the PR list with `.github/scripts/generate-litep2p-changelog.sh`.
5. Run `cargo publish --dry-run` locally to confirm the crate packages.
6. Open a PR to `master`, get it reviewed, and merge.
7. Trigger the Release Publish workflow with the version (step 5 of the automated flow). If that workflow is also broken, a crate owner can tag and publish locally — but publishing should otherwise always go through the `release` environment gate to keep the audit trail.
